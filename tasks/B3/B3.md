Task: B3
## Discover 3 Proactive Security Implementations in Practice

### Description

This task involved researching and documenting three real-world examples of **proactive** security implementations — approaches where organisations anticipate and prevent threats before they materialise, rather than responding to incidents after the fact. Proactive security is defined as security measures that are initiated from within the organisation rather than triggered by an external event. The three implementations below were selected because they represent different layers of proactive security: supply chain/development, continuous external testing, and threat-anticipatory cryptography.

### Findings

#### Finding 1 — DevSecOps: Integrating Security into the Software Development Lifecycle

**What it is:** DevSecOps (Development, Security, and Operations) is the practice of embedding security checks and tools directly into the software development and deployment pipeline, rather than treating security as a separate audit phase at the end of development. It operationalises the principle of "shift left" — moving security review to the earliest stages of development where fixing issues is cheapest and fastest.

**How it works in practice:**
1. **Static Application Security Testing (SAST):** Automated code scanners run on every code commit, checking for known vulnerability patterns (e.g., SQL injection, hardcoded secrets, insecure API calls) before code is merged.
2. **Software Composition Analysis (SCA):** Tools like Dependabot or Snyk scan third-party libraries and open-source dependencies for known CVEs and flag them automatically in pull requests.
3. **Secrets scanning:** Tools scan commits for accidentally committed credentials, API keys, or certificates (e.g., GitHub's built-in secret scanning, GitGuardian).
4. **Infrastructure as Code (IaC) scanning:** Security misconfigurations in cloud infrastructure definitions (Terraform, CloudFormation) are checked before deployment.
5. **Continuous Integration (CI) gates:** Builds fail automatically if critical security issues are detected, preventing vulnerable code from reaching production.

**Real-world evidence:** Intigriti's 2024 vulnerability analysis found that access control flaws and API weaknesses remain the top vulnerability classes precisely because many organisations have not yet integrated RBAC enforcement and API security validation into their CI pipelines. The Action1 cybersecurity predictions for 2025 explicitly identify DevSecOps pipeline integration as the primary mitigation for supply chain and software vulnerability risks. The MITRE ATT&CK framework and VAST threat modelling methodology are increasingly embedded directly into CI/CD pipelines to provide per-sprint threat modelling.

**Why it is proactive:** DevSecOps finds vulnerabilities when the code is first written — before it ever reaches users or attackers. A bug found in development costs roughly 100x less to fix than the same bug found post-breach. It creates a default-secure development culture rather than relying on periodic penetration tests to find issues after the fact.

**Source:** Intigriti — *Software Industry Top Vulnerabilities in 2024*; Action1 — *Top Cybersecurity Vulnerability Predictions for 2025*; ResearchHub — *Master Threat Modeling: 5 Key Frameworks for 2024 Security*.

#### Finding 2 — Continuous Bug Bounty Programs: Always-On Crowd-Sourced Vulnerability Discovery

**What it is:** A bug bounty program pays independent security researchers to find and responsibly disclose vulnerabilities in an organisation's systems. In 2025, the model has matured from occasional engagement into a continuous, always-on practice that runs year-round and is deeply integrated with engineering workflows.

**How it works in practice:**
1. Organisations define a scope (e.g., web application, mobile app, API, cloud infrastructure) and set reward tiers based on severity (e.g., $500 for low, $50,000+ for critical).
2. Researchers independently test within scope and submit detailed vulnerability reports through platforms such as HackerOne, Bugcrowd, or Intigriti.
3. An automated ticketing system routes the report to the relevant engineering team, who triages and patches the issue.
4. After a fix is verified, the researcher receives payment and the vulnerability is marked resolved.
5. Modern programs include direct communication channels between researchers and internal security/engineering teams.

**Real-world evidence:**
- As of 2025, Omdia research confirms that enterprises increasingly recognise bug bounties as essential continuous security mechanisms rather than optional extras.
- Companies now launch bounty programs **before product release**, using them to harden new products proactively rather than waiting for a breach or compliance requirement.
- Bug bounty programs work as a counterincentive to ransomware: rather than attackers being the first to monetise a vulnerability, researchers are financially rewarded to disclose it responsibly.
- The Open Bug Bounty Project demonstrates the model works at the non-commercial level too, with funds saved redirected to remediation.

**Why it is proactive:** Unlike penetration tests (which are scheduled, scoped, and conducted by a small team for a fixed period), bug bounty programs provide continuous, unscheduled coverage by hundreds of researchers with vastly different skills, tools, and attack perspectives. Vulnerabilities are found and fixed before hostile actors have the opportunity to exploit them.

**Source:** Dark Reading — *Bug Bounty Programs Rise as Key Strategic Security Solutions* (2025); Omdia — *Executive Summary: Bug Bounty Programs 2025*; Intigriti — *Top Vulnerabilities in 2024 and What to Watch for in 2025*; Intelligent CISO — *Strengthening Defences Against Ransomware* (2025).

#### Finding 3 — Post-Quantum Cryptography (PQC) Migration: Preparing for a Threat That Has Not Yet Arrived

**What it is:** Post-quantum cryptography (PQC) migration is the proactive replacement of classical cryptographic algorithms (RSA, ECC, Diffie-Hellman) with quantum-resistant alternatives before quantum computers capable of breaking them exist. This is driven by the "harvest now, decrypt later" (HNDL) threat: adversaries are believed to be collecting encrypted data today with the intention of decrypting it once sufficiently powerful quantum computers become available.

**How it works in practice:**
1. **Cryptographic inventory:** Organisations map all systems, libraries, certificates, APIs, and network protocols that use cryptographic functions, identifying where classical algorithms are in use.
2. **Algorithm migration:** NIST finalised three post-quantum algorithms in 2024: ML-KEM (key encapsulation, replacing RSA/ECDH), ML-DSA (signatures, replacing ECDSA), and SLH-DSA (hash-based signatures). Organisations are beginning to replace legacy algorithms in their systems with these approved standards.
3. **Hybrid key exchange:** During the transition, organisations implement hybrid key encapsulation mechanisms (KEM) in TLS 1.3 — combining a classical key exchange with a post-quantum one, so that both need to be broken for the connection to be compromised.
4. **Crypto-agility:** Organisations build policy-driven mechanisms for rapid algorithm swaps, so that if a new vulnerability is discovered in a PQC algorithm, they can migrate quickly without rewriting core systems.

**Real-world evidence:**
- A January 2025 survey of 600 security professionals found that 81% of organisations were actively prototyping or evaluating PQC adoption.
- Thales' 2025 Data Threat Report found that nearly 70% of security leaders cite AI's fast-moving ecosystem as a top threat alongside PQC needs, pointing to crypto-agility as a dual requirement.
- PCI DSS 4.0 (enforceable from 31 March 2025) and EU DORA (in effect from 17 January 2025) now require strong cryptographic governance, accelerating enterprise PQC planning.
- CISA's Zero Trust Architecture Implementation Report (January 2025) explicitly identifies PQC readiness as a component of modern federal security architecture.

**Why it is proactive:** PQC migration addresses a threat that does not yet exist at operational scale. Most organisations with a reactive security posture would not consider acting on a hypothetical future threat. The organisations investing in PQC now are protecting long-lived sensitive data (government records, health records, financial contracts) whose confidentiality must be maintained for decades — well past the point when quantum computing may become practically dangerous. Crypto-agility as a design principle also protects against currently unknown future threats beyond quantum computing.

**Source:** Semnet — *Zero Trust and Encryption Best Practices to Safeguard Data in 2025*; CISA Zero Trust Report (January 2025); Thales 2025 Data Threat Report; NIST post-quantum cryptography standards (2024).

### Reflection

What distinguishes proactive security from strong security (as documented in B2) is intent and timing. A strong security control may still be reactive if it was implemented after a breach or in response to a regulation. Proactive security is defined by the organisation choosing to act before external pressure forces them to.

DevSecOps stood out as the most structurally impactful of the three implementations. By catching vulnerabilities at the point of code creation, it changes the economics of security fundamentally. The cost of a vulnerability that is caught by a SAST scanner during a pull request review is essentially zero compared to the cost of a breach. The barrier to adoption is cultural rather than technical — it requires developers to accept security tooling in their daily workflow, which is why organisational culture and developer training are as important as the tools themselves.

The continuous bug bounty model reflects a growing recognition that no organisation can hire enough internal security talent to keep pace with the global attacker community. Externalising discovery to thousands of independent researchers creates coverage that no internal team can match. The evolution from periodic to always-on programs also reflects a maturation of the model — organisations that only ran a bug bounty when under regulatory scrutiny have learned that attackers do not take time off between audit cycles.

The post-quantum cryptography migration is perhaps the most intellectually striking example because it requires acting on a threat that is probabilistic, future-dated, and technically complex. The organisations doing this work have moved beyond the question "are we secure today?" to "will the decisions we make today still protect data 20 years from now?" That long-term perspective is the defining characteristic of a genuinely proactive security culture.
